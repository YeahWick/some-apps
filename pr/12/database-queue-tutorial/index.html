🗃 Schema Design

Every database queue starts with a well-designed table. The schema must support atomic claiming, state tracking, retries, and ordering.

Why MySQL as a queue? You already have it. Adding RabbitMQ or Redis adds operational burden. For many workloads (< 10k jobs/sec), a well-indexed MySQL table is perfectly sufficient and gives you ACID guarantees for free.

📥 Producing Jobs

Producers insert new jobs into the queue. Jobs can have priorities, scheduled execution times, and arbitrary JSON payloads.

Tip: Use scheduled_at to implement delayed jobs. A job won't be picked up until NOW() >= scheduled_at. This is great for retry backoff or scheduling future work.

🔒 Claiming Jobs: FOR UPDATE SKIP LOCKED

This is the heart of the database queue pattern. SELECT ... FOR UPDATE SKIP LOCKED lets multiple consumers claim jobs without conflicts or blocking.

Without SKIP LOCKED: Consumers would block each other waiting for row locks, creating a bottleneck. With SKIP LOCKED, locked rows are silently skipped — no waiting, no deadlocks.
The atomic claim pattern: Wrap SELECT + UPDATE in a single transaction. The SELECT locks the row, then UPDATE sets state to PROCESSING and records which consumer claimed it. No two consumers can claim the same job.

☵ K8s Consumer Coordination

When multiple Kubernetes pods are consuming from the same queue, coordination happens through the database itself. No external distributed lock needed!

Consumer Identity: Each pod registers itself with a unique consumer_id (typically the pod name from the HOSTNAME env var or metadata.name). This is stored on the job when claimed, so you always know which pod is processing what.

💥 Failure Handling & Retries

Jobs fail. Pods crash. Networks timeout. A robust queue must handle all of these gracefully with retries, exponential backoff, and a dead letter queue.

Visibility Timeout: When a consumer claims a job, set a locked_until timestamp. If the consumer crashes and doesn't complete the job before the timeout, a reaper process will reclaim it. This prevents jobs from being stuck forever.
